# git apply context mismatch when worktree baseline diverges from workspace HEAD

## Context

A² runs candidate edits in a `WorktreeCatalyst` (a `git worktree` branched from a base ref) and then applies the resulting unified diff back to the main workspace via `git apply`. Two independent issues collide here:

1. The benchmark used to call `append_benchmark_test_content()` on the workspace **before** spawning the worktree. The worktree was branched from clean HEAD, so its diff used context lines from clean HEAD. The workspace, however, had the appended test content. `git apply` matches by context line — the context didn't match, and every benchmark patch failed to apply.
2. Once `WorktreeCatalyst::with_base_ref("bench-baseline")` was added so the worktree could pin to a stable tag, the inverse problem appeared: the workspace is always at HEAD, but patches are now baseline-relative. Any file that diverged between baseline and HEAD breaks `git apply` on context.

The general invariant: **`git apply` on a workspace requires that the workspace state at the patched hunks equals the state the patch was generated against**, modulo the changed hunks themselves. Anything else fails noisily.

## Resolution

For the symmetric case (worktree from HEAD, workspace mutated by harness):

```rust
// Revert the workspace to clean HEAD before applying.
if let Err(e) = revert_workspace() {
    eprintln!("[revert before apply failed: {e}]");
}
match try_apply_patch(&patch.diff, &workspace) {
    Ok(true) => {
        // Re-apply harness mutations so post-apply verification can find them.
        if let Err(e) = append_benchmark_test_content(&bench_task) {
            eprintln!("[re-append failed: {e}]");
        }
        // ... run verification ...
    }
    _ => {}
}
```

For the asymmetric case (worktree from baseline tag, workspace at HEAD): there is **no clean fix** that works in-place. The benchmark was redesigned to be observational only (no `--apply`), and promotion is decided by the worktree's own test results.

If applying baseline-relative patches to HEAD is truly required, the only correct approach is: check out the baseline in a temporary index, apply there, then three-way merge / cherry-pick the result onto HEAD. `git apply --3way` is the closest single command and should be tried first.

## Why This Matters

Two non-obvious lessons:

- **`git checkout .` does not "reset to baseline"** when baseline ≠ HEAD. It resets to HEAD. Anyone reasoning about "clean state" in a self-modifying loop must distinguish HEAD-clean from baseline-clean.
- **Mutating the workspace during a worktree-based evaluation breaks the diff symmetry silently**. The worktree has no idea the workspace moved underneath it, and patch context errors are notoriously cryptic.

The deeper architectural takeaway: **evaluation should never mutate the substrate it evaluates against**. The fix that finally stuck was not clever git plumbing — it was deleting `--apply` entirely and treating the benchmark as observation.

## Examples

Failure mode (silent until apply):
```
worktree branched from: HEAD (clean)
workspace state:        HEAD + appended test fn
patch context:          clean HEAD lines
git apply target:       workspace (mutated)
result:                 error: patch failed: src/lib.rs:42
                        error: src/lib.rs: patch does not apply
```
