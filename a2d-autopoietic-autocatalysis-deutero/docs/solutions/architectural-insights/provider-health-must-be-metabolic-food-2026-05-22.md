# Provider Health Must Be Metabolic Food

## Problem

A²D was still relying on a human operator to read run logs and patch provider assignment/fallback policy. That violates the intent of an autopoietic autocatalytic system: provider-role degradation is part of the environment, so it must enter the metabolism as typed food.

Recent live runs made this obvious:

- coder/evolver timeouts required manual provider reassignment;
- topology comparison lineage exposed `evolver` falling back to GLM;
- the fix was made manually in Rust rather than by the cycle observing and adapting to provider-role evidence.

## Fix

The metabolism now produces a mechanical `provider_health_report` artifact at cycle end.

The artifact contains JSON with:

- `cycle`;
- currently unavailable/cooled-down providers;
- provider health counters (`consecutive_failures`, `cooling_down`);
- recent invocation lineage: enzyme, provider, outcome, error, and coder candidate portfolio records.

This report is inserted into the artifact graph by the metabolism, not generated by a model.

The live seed/loaded germlines now include `provider_health_report` as food. It is also routed as a catalyst to:

- `evolver` — so enzyme definition adaptation sees provider-role failure evidence;
- `architect` — so self-modification prompts can reason about orchestration/provider bottlenecks.

Specialized evolver and architect prompts now include a provider health section when available.

## Why this matters

This does not yet make provider assignment fully evolvable. It closes the missing information edge first:

```text
provider failures/timeouts → provider_health_report → evolver/architect
```

Before this, the only actor with provider-health information was the human reading logs. Now the system has typed internal evidence it can metabolize.

This is the autopoietic direction: the observer produces food; the catalytic network consumes it; future mutations/patches can be gated mechanically.

## Next step

Make provider policy itself an evolvable/gated mechanism, not just provider health as feedback. A minimal next slice:

1. define a typed `provider_policy` artifact;
2. allow architect/evolver to propose provider-role policy changes;
3. self-sandbox/gate them with bounded topology comparison;
4. commit accepted policy into lineage.
